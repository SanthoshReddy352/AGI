  The core problem

  Your "semantic recall" is not semantic. MemoryStore persists to Chroma, but the
  embedding function (HashEmbeddingFunction, line 41) hashes tokens into 64 buckets via
   SHA-256. Two paraphrases that share no literal tokens get near-orthogonal vectors —
  so "what's my sister's name" won't recall "my sibling is called Asha." The SQL
  fallback (_rank_candidates) is literal token-overlap counting. Both layers of the
  primary memory RAG are keyword matching wearing a vector-store costume.

  Meanwhile, a genuinely good local embedder (BGESmallEmbedder) already exists in the 
  codebase — it's just wired only into plan_archive.py, not the main recall path.

  Gaps, ranked

  1. 🔴 Placeholder embedder — the central issue above. The fix exists in-repo but
  isn't used.
  2. 🔴 Silent permanent degradation — memory_store.py:171 sets _vector_available = 
  False on any Chroma exception and never retries. One transient error downgrades the
  whole process to token-fallback for its lifetime.
  3. 🟠 Model fragmentation — 3 embedders, 2 dims, no shared instance → wasted RAM and
  inconsistent quality across subsystems.
  4. 🟠 No hybrid fusion — dense (Chroma) and sparse (FTS5) retrieval exist side by
  side but are never fused (e.g. RRF). Hybrid is the production default.
  5. 🟠 No reranking / MMR — raw top-k cosine only; no cross-encoder rerank, no
  diversity.
  6. 🟡 No retrieval eval harness — no recall@k metric, no regression test. You can't
  tell if a change helps or hurts.
  7. 🟡 Coarse filtering — vector query filters only on session_id; no time decay,
  persona scope, or memory_type weighting. (Cross-session recall is a listed feature
  but the filter doesn't support it well.)
  8. 🟡 Migration hazard — swapping the 64-dim hash for a 384-dim model requires
  re-indexing the existing Chroma collection (keyed to friday-hash-v1).
  9. 🟢 Minor — no embedding cache; fixed 600-char chunks with no overlap in
  SessionRAG; BM25 b=0.0 disables length normalization.

  There's real scope for improvement. Before I draft the implementation plan, I need to
   pin down a few decisions that change the design substantially.

  Plan — RAG production-hardening (8 tracks)

  Track 1 — Real semantic memory recall + unified embedder (foundation)
  Stand up a process-wide get_shared_embedder() singleton (default all-MiniLM-L6-v2 —
  already loaded by the router, so no new model in RAM, configurable in config.yaml).
  Add a Chroma EmbeddingFunction that delegates to it, named friday-st-v1. MemoryStore
  uses it when sentence-transformers is present, keeps HashEmbeddingFunction only as
  the offline fallback. Point plan_archive.py at the same singleton.

  Track 2 — Resilient vector store
  Replace the permanent _vector_available=False kill-switch with consecutive-failure +
  cooldown retry, so one transient Chroma error doesn't downgrade the whole process.

  Track 3 — Lazy re-index migration
  Detect the 64→384 dim / embedding-fn-name mismatch on the existing friday_memory
  collection, drop & recreate it. Old memories repopulate as written (your "lazy"
  choice). I'll also add an optional reindex_memory() you can trigger manually if you
  want a full backfill.

  Track 4 — Hybrid retrieval (RRF fusion)
  Fuse dense (Chroma) + sparse (FTS5 over turns) results via Reciprocal Rank Fusion in
  semantic_recall.

  Track 5 — Rerank + MMR diversity
  MMR over fused candidates (reuses embeddings, zero extra model). Optional lazy
  cross-encoder reranker behind a config flag.

  Track 6 — Smarter filtering
  Recency time-decay, persona-scope boost, and memory_type weighting applied to
  scoring.

  Track 7 — Embedding cache
  Content-hash LRU cache to avoid re-embedding identical text.

  Track 8 — Eval harness
  tests/retrieval/ with a labeled query→memory set computing recall@k / MRR, plus a
  pytest gate so future changes can't silently regress quality.

  Each track ships with tests (per your intent-pattern/test rules) and a
  docs/testing_guide.md update where behavior is user-visible.
